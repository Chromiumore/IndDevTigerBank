# Кратко опишите предметную область: что моделирует программа
Программа моделирует систему учёта финансов банка.

# Используйте единый code style
 * Названия классов в PascalCase
 * Названия абстрактных классов содержат в себе слово "Base"
(избегается использование слов "Abstract" в названии абстрактных классов,
использования названий по типу IService для интерфейсов, ServiceImp для
реализаций интерфейсов)

# Примените паттерн Фабричный метод
Созданы фабрики для всех доменных сущностей

# Примените паттерн Фасад
 * Созданы сервисы для управления доменными сущностями
 * Ещё с предыдущего дз в проекте реализован сервис для экспорта данных, выполняющий роль фасада

# Примените паттерн Шаблонный метод
Паттерн используется в стратегиях импорта и экспорта данных (Например: метод writeFile в стратегии вывода является шаблонным методом,
использующим другие методы, которые должны быть определены в классах-наследниках)

# Добавьте модульное теcтировании (в том числе и реализации паттернов)
Были добавлены unit- и интеграционные тесты:
 * Тест фабрик сущностей
 * Тест сервисов импорта и экспорта
 * Тесты для сервисов сущностей

# Соблюдайте принципы SOLID
## Single Responsibility
Каждый сервис отвечает за свою доменную область
## Open/Closed
Стратегии экспорта позволяют добавлять новые форматы
## Liskov Substitution
Все наследники BaseRepository могут использоваться везде, где ожидается BaseRepository
## Interface Segregation
Репозитории содержат только нужные им методы
## Dependency Inversion
Родительский класс репозиториев зависит от абстрактного класса сущности, а не от
конкретной реализации

# Соблюдайте принципы GRASP
Примеры соблюдения Low Coupling:
 * Использование стратегий для импорта и экспорта. Сервисы зависят от абстракций
 * Сервисы, использующие фабрики, не знают, как именно создаются объекты в них
 * Репозитории изолируют сервисы от логики работой с БД (с HashMap в роли заглушки)
Примеры соблюдения High Cohesion:
 * Доменные сущности - данные и бизнес-логика
 * Фабрики - создание сущностей
 * Репозитории - хранение и изменение объектов
 * Сервисы - сложная бизнес-логика приложений

# Реализуйте взаимодействие модулей через DI-контейнер
Spring управляет жизненным циклом объектов и их зависимостями.
Это упрощает тестирование и делает связи между классами явными.
Примеры: 
 * Внедрение зависимостей через конструктор класса AnalyticsService
 * Внедрение зависимостей через поля с аннотацией @Autowired в тестах и сервисах

# Реализуйте CRUD операции для счетов, категорий, операций (доходов/расходов)
CRUD-операции реализованы на двух уровнях:
 * Репозитории сущностей (с заглушкой для БД)
 * Сервисы сущностей, которые используют репозитории и дополняют их операции новой логикой

# Реализуйте все функциональные требования
Были реализованы следующие требования:
 * Расчет финансов
 * Аналитика
 * Импорт и экспорт данных
 * Статистика (время выполнения метода)

# Укажите ситуации, при которых появятся проблемы добавления нового функционала или расширении бизнес-логики
 * Сильная связь с типом данных в стратегиях экспорта (Проверка типа сущности для определения заголовков)
 * Тестирование при добавлении новых сущностей (код тестов дублируется)
 * Добавление новых стратегий импорта и экспорта данных (в т.ч. дублирование существующего кода)

# Приведите аргументы, почему введенные абстракции улучшили качество дизайна
 * Каждый класс имеет единственную причину для изменения.
 Классы становятся меньше, и легче тестируются.
 * Паттерн "стратегия" позволяет легко добавлять новые форматы экспорта
 * Инкапсуляция логики в сущностях (Это распространённая практика,
которая упрощает реализацию бизнес-логики. Это фактически отличает
доменную сущность от моделей данных и DTO)
 * Использование DI-контейнера (описано выше)
 * Сервисы-фасады скрывают сложную логику и взаимодействие разных модулей, предоставляя удобный интерфейс
 * Фабрики инкапсулируют логику создания объектов. В будущем может добавляться
более сложная логика создания объектов, не затрагивая при этом классы создаваемых сущностей
 * Шаблонные методы для импорта и экспорта переложили ответственность за детали алгоритма на подклассы.
Общий алгоритм импорта и экспорта при этом не меняется.
 * Модульные тесты позволяют выявить неочевидные с первого взгляда ошибки в реализации программных модулей.
